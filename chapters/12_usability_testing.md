# 第 12 章：可用性测试

> 目标：掌握可用性测试的方法，学会收集有效用户反馈，用数据驱动设计迭代。

---

## 12.1 什么是可用性测试

可用性测试是观察真实用户使用产品完成特定任务的过程，目的是发现设计中的问题。

> "如果你观察一个用户，你发现一个问题；如果你观察五个用户，你发现 85% 的问题。"
> —— 雅各布·尼尔森

### 可用性测试 vs 用户研究

| 对比 | 可用性测试 | 用户研究（访谈） |
|------|-----------|----------------|
| 目的 | 发现设计问题 | 理解用户需求和行为 |
| 时机 | 设计阶段和之后 | 设计之前 |
| 内容 | 让用户完成任务 | 问用户问题 |
| 产出 | 可用性问题列表 | 用户洞察 |

---

## 12.2 可用性测试的三种类型

### 1. 调节式测试（Moderated Testing）

主持人在场，观察用户操作并可以追问。

```
优点：可以深入追问，获取更多信息
缺点：主持人可能无意影响用户行为，成本较高

适合：
  • 探索性测试（产品初期）
  • 需要深度理解用户思路
  • 复杂任务的测试
```

### 2. 非调节式测试（Unmoderated Testing）

用户独立完成测试，工具自动录制。

```
优点：成本低，可同时测试更多用户，用户在自然环境中操作
缺点：无法追问，无法处理突发情况

工具：UserTesting.com、Maze、Useberry

适合：
  • 验证性测试（设计中后期）
  • 简单明确的任务
  • 需要快速获得大量数据
```

### 3. 游击式测试（Guerrilla Testing）

在公共场所（咖啡馆、图书馆）随机邀请路人参与测试。

```
优点：零成本，快速，适合早期验证
缺点：参与者不一定是目标用户

适合：
  • 预算极有限时
  • 验证最基础的可用性假设
  • 快速迭代的早期阶段
```

## 12.3 测试准备

### 确定测试目标

**好的测试目标**（具体、可验证）：
```
✓ 用户能否在 2 分钟内找到退款入口？
✓ 用户是否能理解"智能推荐"功能的工作方式？
✓ 新用户注册时，哪一步骤流失率最高？
```

**坏的测试目标**（模糊）：
```
✗ 了解用户对我们 App 的看法
✗ 用户喜不喜欢新设计
```

### 设计测试任务

**任务设计原则**：

1. **现实场景，不是操作指令**
```
✗ 错误（操作指令）："请点击右上角的购物车图标"
✓ 正确（现实场景）："你想买一双运动鞋，请完成购买流程"
```

2. **可观察、可判断完成与否**
```
✓ 明确的完成标准："成功下单并看到订单确认页"
✗ 模糊："探索一下这个 App"
```

3. **从用户目标出发，而非功能名称**
```
✗ 错误："请找到'个人中心'页面"（暴露了功能名称）
✓ 正确："你想修改你的收货地址，请操作"
```

### 测试脚本模板

```markdown
## 可用性测试脚本

### 开场（5分钟）
感谢参与。今天不是测试你，是测试设计——没有对错，
你的自然操作对我们最有价值。

操作时请大声说出你的想法（出声思考法）。
我们会录音/录屏，只用于内部研究。

有问题我不会直接告诉你答案，但测试结束后可以解答。

### 背景问题（5分钟）
- 你平时用哪些类似的 App？
- 你多久使用一次 [相关行为]？

### 任务 1（10分钟）
场景：[描述真实场景]
任务：[具体目标]
观察记录：操作路径 / 卡顿点 / 误操作 / 口头反馈

### 任务 2...（重复）

### 收尾（10分钟）
- 整体感受如何？
- 哪里最困惑？
- 哪里最顺畅？
- 还有什么想说的？
```

## 12.4 测试执行

### 角色分工

```
主持人（Moderator）：
  • 引导用户完成任务
  • 保持中立，不引导不评判
  • 在合适时机追问

观察者（Observer）：
  • 记录用户行为和语言
  • 注意非语言信号（皱眉、犹豫）
  • 不发言，不干预

记录员（Note-taker）：
  • 详细记录操作步骤
  • 标注时间戳
  • 记录关键引言
```

### 出声思考法（Think Aloud Protocol）

引导用户说出他们的思考过程：

```
"请一边操作，一边说出你在想什么，
 你看到什么，你在期待什么，
 你感到困惑的地方也请说出来"

追问话术（当用户沉默时）：
  • "你刚才在想什么？"
  • "你现在期望发生什么？"
  • "这里让你困惑了吗？"
```

### 保持中立

```
✗ 绝对不能说：
  "大多数用户都能在这里找到..."
  "这个按钮其实是..."
  "没关系，这是很多人都会遇到的问题"

✓ 保持中立的话术：
  "有意思，你为什么这样操作？"
  "你现在的想法是？"
  "如果继续操作，你接下来会怎么做？"
```

## 12.5 数据分析

### 定性数据分析：亲和图（Affinity Diagram）

1. 把每个观察、用户引言、问题写在便利贴上
2. 将相似的便利贴分组
3. 为每组起一个概括性标题
4. 找出高频问题（出现在多个用户中的问题）

```
分组示例：
  [导航困惑]           [支付流程]          [信息过载]
  • "找不到购物车"      • "不知道付款成没"   • "太多选项了"
  • "以为要点那个图标"  • "等了很久没反应"   • "界面太复杂"
  • "返回后找不到了"                        • "信息太密了"
```

### 可用性问题评级

对每个发现的问题进行严重程度评级：

| 评级 | 描述 | 处理优先级 |
|------|------|---------|
| **P0 - 阻断性** | 用户无法完成任务（任务失败率 > 50%） | 必须修复再发布 |
| **P1 - 严重** | 用户完成任务困难，或有重大困惑 | 尽快修复 |
| **P2 - 中等** | 影响效率或体验，但可以完成任务 | 下个版本修复 |
| **P3 - 轻微** | 小问题，极少数用户遇到 | Backlog 中跟踪 |

### 定量指标

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| **任务成功率** | 成功完成任务的用户比例 | 成功人数/总人数 |
| **任务完成时间** | 完成任务所用时间 | 平均时间（去掉极值）|
| **错误率** | 完成任务中发生的错误次数 | 平均错误次数/人 |
| **满意度（SUS）** | 系统可用性量表 | 10题问卷，0-100分 |

**SUS 评分参考**：
```
> 90：优秀
80-90：好
70-80：一般
< 70：需要改进
< 51：不及格
```

## 12.6 测试报告

### 报告结构

```
可用性测试报告
├── 1. 测试概述
│   ├── 测试目标
│   ├── 测试时间/人数
│   └── 测试方式
├── 2. 关键发现（执行摘要）
│   └── Top 5 最重要的问题
├── 3. 详细问题清单
│   ├── P0 问题 + 截图/录像时间戳 + 建议方案
│   ├── P1 问题...
│   └── P2 问题...
├── 4. 正面发现（哪些做得好）
├── 5. 数据汇总
│   ├── 任务完成率表格
│   └── SUS 评分
└── 6. 下一步建议
```

### 从发现到解决方案

```
问题描述（What）：
  "3/5 的用户找不到'修改地址'的入口，在个人中心页面停顿超过 30 秒"

原因分析（Why）：
  "用户期望地址管理在'我的订单'中，
   而我们放在了'账户设置'的深层级中"

设计建议（How）：
  1. 快速修复：在订单页面添加地址编辑的快捷入口
  2. 根本修复：重新设计地址管理的信息架构
```

## 12.8 持续测试文化

### 何时进行测试

```
• 设计概念验证：有了低保真原型时（越早越好）
• 设计方案对比：有 A/B 两个方案时
• 重大改版前：发布前最后验证
• 发布后：收集真实用户数据
• 定期测试：每季度一次，持续监测
```

### 快速迭代循环

```
设计 → 测试 → 分析 → 修改 → 再测试
 ↑                              |
 └──────────────────────────────┘

目标：每次迭代解决最高优先级问题，
      不要等到"完美"再测试，
      测试发现问题正是测试的价值所在
```

## 本章实践作业

**作业 1**：为你设计的 App 原型，设计一个可用性测试方案：
- 确定 3 个测试目标
- 设计 3 个测试任务（现实场景描述）
- 写出完整的测试脚本

**作业 2**：找 2 位朋友，用你设计的 Figma 原型进行 30 分钟可用性测试：
- 录屏（需征得同意）
- 使用出声思考法
- 记录所有卡顿点和困惑

**作业 3**：整理测试发现，制作一份简单的测试报告：
- 列出所有发现的问题
- 按严重程度（P0-P3）分类
- 为每个 P0/P1 问题提出设计建议

## 本章小结

| 概念 | 要点 |
|------|------|
| 测试类型 | 调节式/非调节式/游击式，各有适用场景 |
| 任务设计 | 现实场景，不是操作指令，有明确完成标准 |
| 出声思考 | 引导用户说出思考过程，保持中立 |
| 问题分级 | P0阻断→P1严重→P2中等→P3轻微 |
| 持续测试 | 测试不是一次性的，要融入设计流程 |
