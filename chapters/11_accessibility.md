# 第 11 章：无障碍设计（A11y）

> 目标：理解无障碍设计的标准和实践，让产品对所有人都友好可用。

---

## 11.1 什么是无障碍设计

**无障碍设计（Accessibility，简写 A11y）** 指确保产品对不同能力的人都可以使用，包括有视觉、听觉、运动、认知障碍的用户。

> "A11y" 是 Accessibility 的数字缩写：A + 11个字母 + y

### 为什么需要无障碍设计

**数字**：
- 全球约 **15%** 的人口（超过 10 亿人）有某种形式的残障
- 中国有约 **8500 万** 残障人士
- 老年人（60岁+）通常有视力下降、手指灵敏度降低等问题

**好处**：
- 无障碍设计对所有用户都有益（比如字幕对听障和噪音环境都有用）
- 改善 SEO（语义化 HTML 对搜索引擎友好）
- 部分国家和地区有法律要求（如美国 ADA 法案）
- 体现企业社会责任和品牌价值

## 11.2 WCAG 无障碍标准

**Web Content Accessibility Guidelines（WCAG）** 是国际通行的无障碍标准，当前版本为 WCAG 2.1。

### 四大原则（POUR）

| 原则 | 含义 |
|------|------|
| **P - 可感知（Perceivable）** | 信息能被用户感知，不依赖单一感官 |
| **O - 可操作（Operable）** | 界面可以通过多种方式操作 |
| **U - 可理解（Understandable）** | 内容和操作容易理解 |
| **R - 健壮性（Robust）** | 内容被多种技术（辅助技术）解析 |

### 符合级别

```
A 级：最低要求，基本无障碍
AA 级：标准要求，大多数法规要求达到此级别
AAA 级：最高标准，少数场景可达
```

## 11.3 视觉无障碍

### 颜色对比度（回顾第7章）

```
WCAG AA 标准：
  普通文字（< 18px）：  对比度 ≥ 4.5:1
  大文字（≥ 18px Bold）：对比度 ≥ 3:1
  非文字元素（图标、边框）：对比度 ≥ 3:1

快速验证工具：
  • Figma 插件：A11y - Color Contrast Checker
  • WebAIM Contrast Checker（在线工具）
  • Stark 插件（Figma/Sketch）
```

**常见失败案例**：
```
浅灰文字在白底：#999999 对比度 2.85:1  ✗ 不通过
占位符文字太浅：#BBBBBB 对比度 1.73:1  ✗ 不通过
蓝色链接在白底：#0066CC 对比度 7.0:1   ✓ 通过
```

### 不仅靠颜色传递信息

```
✗ 错误：表单只用红色边框表示出错
✓ 正确：红色边框 + 错误图标 + 文字说明

✗ 错误：图表只用颜色区分不同数据系列
✓ 正确：颜色 + 不同的图案/形状（供色盲用户区分）

✗ 错误：必填项只用红色 * 标注
✓ 正确：红色 * 和文字"必填"说明
```

### 文字大小与缩放

```
• 正文最小字号：16px（Web），13px（移动端）
• 支持系统字体缩放：用户设置大字体时，界面不应错乱
• 避免 CSS 使用 px 锁定字号（改用 rem，随用户设置缩放）
• 避免固定高度容器（文字变大时不应溢出）
```

## 11.4 键盘导航

大量用户（尤其是运动障碍用户）使用键盘而非鼠标操作界面。

### 键盘导航基本规则

**Tab 键顺序**：
```
用户按 Tab 键时，焦点应该按逻辑顺序移动（通常从上到下，从左到右）

✗ 错误：Tab 顺序随机跳转，用户不知道焦点在哪里
✓ 正确：Tab 顺序与视觉顺序一致，焦点清晰可见
```

**可见的焦点指示器（Focus Indicator）**：
```
当用户用键盘聚焦到元素时，必须有明显的视觉指示：

✗ 错误：outline: none; （删掉默认焦点环）
✓ 正确：为所有可交互元素设计焦点样式

示例焦点样式：
  outline: 2px solid #4299E1;
  outline-offset: 2px;
```

**常用键盘操作**：

| 键 | 操作 |
|----|------|
| Tab | 移动到下一个可交互元素 |
| Shift + Tab | 移动到上一个可交互元素 |
| Enter / Space | 激活按钮/链接/复选框 |
| ↑↓ 箭头 | 在列表/菜单中导航 |
| Escape | 关闭弹窗/菜单 |

### 焦点管理

**弹窗打开时**：
```
1. 焦点自动移入弹窗内（通常移到标题或第一个输入框）
2. 焦点被困在弹窗内（Tab 不会跳到背景元素）
3. 关闭弹窗时，焦点返回触发弹窗的元素
```

**模态弹窗（Modal）键盘陷阱**：
```
这是要求，不是 Bug！
弹窗打开时，Tab 循环仅在弹窗内部，
防止用户意外操作背景内容
```

## 11.5 屏幕阅读器

屏幕阅读器（Screen Reader）将屏幕内容转化为语音，供视觉障碍用户使用。

### 主流屏幕阅读器

| 平台 | 屏幕阅读器 | 常用快捷键 |
|------|-----------|---------|
| iOS | VoiceOver | 三指轻点开启 |
| Android | TalkBack | 音量键+下键 3秒 |
| macOS | VoiceOver | Cmd+F5 |
| Windows | NVDA（免费）、JAWS | Insert+Q |

### 语义化与 ARIA

**设计师需要了解的概念**（告知开发如何实现）：

**语义化 HTML**：
```html
✗ 错误（没有语义）：
  <div onclick="submit()">提交</div>

✓ 正确（有语义）：
  <button type="submit">提交</button>

语义化元素让屏幕阅读器知道这是"按钮"，
可以用 Enter 键触发
```

**ARIA 标签**：
```
aria-label：为没有可见文字的元素提供说明
  示例：图标按钮 <button aria-label="删除此项"> × </button>

aria-describedby：关联额外描述信息
  示例：输入框关联错误提示

alt 属性（图片替代文字）：
  <img alt="产品主图：白色无线耳机">

  ✗ 错误：alt="" （装饰性图片可以为空，但内容图片不行）
  ✗ 错误：alt="image123.jpg" （不要用文件名）
  ✓ 正确：alt="用户点击购物车按钮的截图"
```

### 阅读顺序

屏幕阅读器按 DOM 顺序（代码顺序）读取，不是视觉顺序。

```
✗ 错误：CSS 用 position:absolute 改变了视觉位置，
       但 DOM 顺序不对，阅读器读出顺序与视觉不符

✓ 正确：保证 HTML 结构顺序与视觉阅读顺序一致
```

## 11.6 运动障碍无障碍

### 触控目标大小

```
iOS HIG：最小触控目标 44×44pt
Material Design：最小触控目标 48×48dp
WCAG 2.5.5（AAA）：最小 44×44px

对于小图标：可以用 padding 扩大点击区域：

┌──────────────────────────┐  ← 实际可点击热区：48×48
│                          │
│     ┌────────┐           │
│     │  图标  │           │  ← 视觉图标：24×24
│     └────────┘           │
│                          │
└──────────────────────────┘
```

### 减少运动（Reduce Motion）

部分用户（如有前庭障碍的人）对过度动效会产生恶心/不适：

```css
/* 媒体查询：用户开启了减少动效 */
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

**设计实践**：重要动效应有非动效替代版本（如直接显示/隐藏）

## 11.7 认知无障碍

### 简单语言

```
避免：使用专业术语、复杂长句
推荐：
  • 句子简短（每句 ≤ 20 字）
  • 使用主动语态
  • 避免双重否定
  • 用日常词汇

✗ 错误："因系统检测到账户存在异常安全活动，您的访问权限已被暂时中止。"
✓ 正确："您的账户已被锁定。请联系客服解锁。"
```

### 一致性

```
• 相同功能，始终用相同的名称（不要"删除"和"移除"混用）
• 导航结构在所有页面保持一致
• 图标含义在整个产品中保持一致
```

### 错误预防与清晰的错误信息

```
预防错误：
  • 输入格式提示放在输入框下方（不要等提交后才显示）
  • 日期选择器比文本输入更防错
  • 下拉选择比开放输入更防错

清晰的错误信息：
  ✗ 错误："输入无效"
  ✓ 正确："手机号格式不正确，请输入 11 位数字（如：138XXXXXXXX）"

  错误信息要告诉用户：
  1. 出了什么问题
  2. 为什么出了问题（如果不明显）
  3. 如何修复
```

## 11.8 无障碍设计检查清单

### 视觉

- [ ] 所有文字对比度 ≥ 4.5:1（普通文字）
- [ ] 不仅靠颜色传递重要信息
- [ ] 图片都有有意义的 alt 文字
- [ ] 支持文字缩放（不破坏布局）
- [ ] 正文字号 ≥ 16px（Web）/ 13px（App）

### 交互

- [ ] 所有功能可用键盘完成
- [ ] 焦点指示器清晰可见
- [ ] Tab 顺序合理
- [ ] 弹窗打开时焦点正确管理
- [ ] 触控目标 ≥ 44×44px

### 内容

- [ ] 使用简单语言，避免术语
- [ ] 错误信息具体且可操作
- [ ] 表单控件都有可见标签（不只是 placeholder）
- [ ] 页面有清晰的标题结构（H1→H2→H3）

### 媒体

- [ ] 视频有字幕
- [ ] 纯音频内容有文字稿
- [ ] 闪烁内容频率 < 3 次/秒（防止癫痫发作）

## 本章实践作业

**作业 1**：用手机开启 VoiceOver（iOS）或 TalkBack（Android），盲操一款你熟悉的 App 5 分钟，记录：
- 哪些操作顺畅？
- 哪些地方屏幕阅读器无法正确描述内容？
- 你认为设计师需要如何改进？

**作业 2**：检查你之前设计的界面：
- 用 Figma 的 Contrast Checker 插件检查所有文字对比度
- 检查所有可点击元素的大小是否 ≥ 44px
- 找出任何仅靠颜色传递信息的地方并修改

**作业 3**：重新设计之前设计的一个错误提示状态，确保：
- 不仅靠红色（颜色）表示错误
- 错误信息告诉用户如何修复
- 对比度达标

## 本章小结

| 概念 | 要点 |
|------|------|
| WCAG 标准 | 可感知/可操作/可理解/健壮性，AA 级为基本目标 |
| 颜色对比度 | 普通文字 ≥ 4.5:1，不能仅靠颜色传递信息 |
| 键盘导航 | 所有功能可用键盘完成，焦点指示器必须可见 |
| 屏幕阅读器 | 语义化结构、alt 文字、ARIA 标签 |
| 触控目标 | 最小 44×44px |
| 认知无障碍 | 简单语言、一致性、具体的错误信息 |
