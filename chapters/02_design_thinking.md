# 第 02 章：设计思维与用户研究

> 目标：掌握设计思维框架，学会用研究方法理解用户，而不是依赖假设。

---

## 2.1 设计思维（Design Thinking）

设计思维是一种以人为中心的问题解决框架，由斯坦福 d.school 推广。

### 五步模型

```
  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  │  共情    │→  │  定义    │→  │  发散    │→  │  原型    │→  │  测试    │
  │ Empathize│   │  Define  │   │  Ideate  │   │Prototype │   │   Test   │
  └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
  理解用户真实     提炼核心问题    头脑风暴方案    快速制作原型    验证假设
  需求与痛点       陈述            不设限制        低成本试错      收集反馈
```

> 注意：这五步不是线性流程，而是**迭代循环**。随时可以回到前一步。

### 双钻模型（Double Diamond）

英国设计委员会提出的另一个经典框架：

```
    发散                收敛                发散                收敛
  ●─────────────●    ●─────────────●
 /   探索问题   \  /   探索方案   \
●                ●                 ●
         ↑                ↑
      发现真正的问题      找到正确的解决方案
```

- **第一个钻石**：深入理解问题（不要急于解决）
- **第二个钻石**：探索并验证解决方案

## 2.2 用户研究方法概览

```
定性研究（Why/How）          定量研究（How many/What）
─────────────────            ─────────────────────────
• 用户访谈                   • 问卷调查
• 焦点小组                   • A/B 测试
• 民族志观察                 • 数据分析（埋点）
• 可用性测试                 • 眼动追踪
• 日记研究                   • 热图分析

→ 理解用户行为背后的原因      → 了解行为的规模和频率
```

**原则**：定性与定量结合，互相验证。

## 2.3 用户访谈

用户访谈是最重要的定性研究方法，直接听用户说话。

### 访谈前准备

**1. 确定研究目标**
不要问"用户喜不喜欢我们的产品"，而是问：
- 用户现在如何完成[某个任务]？
- 他们在这个过程中遇到什么障碍？
- 他们如何评估一个好的[产品/服务]？

**2. 筛选受访者**
- 选择目标用户，而非随机用户
- 每轮研究访谈 5-8 人（超过 5 人能发现约 85% 的问题）
- 避免选择同事或朋友（认知偏差）

**3. 设计访谈提纲**

```
访谈提纲结构：
├── 开场（5 分钟）：建立信任，说明录音/记录
├── 背景了解（10 分钟）：用户基本情况、使用习惯
├── 核心问题（20-30 分钟）：围绕研究目标深挖
├── 任务观察（可选）：让用户演示实际操作
└── 收尾（5 分钟）：有无补充，感谢
```

### 好问题 vs 坏问题

| 坏问题（避免） | 好问题（推荐） |
|-------------|-------------|
| "你喜欢我们的 App 吗？" | "跟我说说你上次用[类似产品]的经历" |
| "你会用这个功能吗？" | "你现在是怎么处理[这个问题]的？" |
| "这个设计好不好？" | "你看到这个页面时，第一反应是什么？" |
| "你需要搜索功能吗？" | "你找不到某样东西时，会怎么做？" |

### 访谈技巧

- **5 个 Why**：不断追问"为什么"，挖到根因
- **沉默是金**：用户停顿时别急着填补，等待更多信息
- **不要引导**：避免说"你是不是觉得……"
- **关注行为，不是观点**：问"你上次怎么做的"而非"你一般会怎么做"
- **同理心**：不评判用户的行为，保持中立好奇

## 2.4 问卷设计

问卷适合大规模收集数据，验证定性研究的发现。

### 问卷设计原则

**1. 每个问题只问一件事**
```
✗ 错误："这个产品是否实用且美观？"
✓ 正确：分拆为两个问题分别问
```

**2. 避免引导性措辞**
```
✗ 错误："你是否觉得我们的客服很糟糕？"
✓ 正确："请评价你对客服的满意度（1-5分）"
```

**3. 量表题使用规范**

| 量表类型 | 用途 | 示例 |
|---------|------|------|
| Likert 5分量表 | 态度、满意度 | 非常不同意→非常同意 |
| NPS（净推荐值） | 用户忠诚度 | 0-10 推荐可能性 |
| 语义差异量表 | 品牌感知 | 复杂←——→简单 |
| SUS 系统可用性量表 | 产品易用性 | 10题标准化问卷 |

**NPS 计算方式**：
```
推荐者（9-10分）的比例 - 批评者（0-6分）的比例 = NPS
NPS > 50 = 优秀  |  NPS > 0 = 良好  |  NPS < 0 = 需改进
```

## 2.5 竞品分析

### 如何进行竞品分析

**第一步：确定竞品范围**

```
直接竞品          间接竞品          参考竞品
────────          ────────          ────────
同类型产品         解决同类需求       体验优秀的
（功能相似）       的不同形态         行业标杆
例：美团 vs 饿了么  例：外卖 vs 便利店  例：Airbnb 的
                                    预订流程
```

**第二步：分析维度**

| 维度 | 分析内容 |
|------|---------|
| 核心功能 | 提供哪些功能？如何分层？ |
| 信息架构 | 导航结构、页面层级 |
| 交互流程 | 关键任务的步骤数、流程顺畅度 |
| 视觉风格 | 色彩、字体、组件风格 |
| 差异化亮点 | 独特功能或体验优势 |
| 痛点与不足 | App Store 差评、用户吐槽 |

**第三步：整理对比矩阵**

```markdown
| 功能/特性        | 竞品 A | 竞品 B | 竞品 C | 我们 |
|----------------|--------|--------|--------|------|
| 注册流程步骤数   |   3    |   5    |   2    |  ?   |
| 是否支持第三方登录|   ✓   |   ✓   |   ✗    |  ?   |
| 新手引导        |   有   |   无   |   有   |  ?   |
| 深色模式        |   ✓   |   ✗   |   ✓    |  ?   |
```

## 2.6 用户画像（Persona）

用户画像是基于研究数据构建的虚构用户原型，帮助团队对齐"我们在为谁设计"。

### 用户画像模板

```
┌─────────────────────────────────────────────────────┐
│  [头像]   小雨，28岁，上海，产品经理                 │
│           "我需要快速找到可靠的解决方案"              │
├─────────────────────────────────────────────────────┤
│  基本信息                                            │
│  • 工作：互联网公司产品经理，工作3年                  │
│  • 设备：iPhone 15、MacBook Pro                     │
│  • 技术水平：中等，常用各类 App                      │
├─────────────────────────────────────────────────────┤
│  目标（Goals）                                       │
│  • 快速完成日常工作任务                               │
│  • 减少重复低效的工作                                 │
│  • 在同事面前展示专业能力                             │
├─────────────────────────────────────────────────────┤
│  痛点（Pain Points）                                  │
│  • 工具太多，信息分散，切换成本高                     │
│  • 不确定数据是否准确，需要反复核实                   │
│  • 没时间学习复杂的新工具                             │
├─────────────────────────────────────────────────────┤
│  行为（Behaviors）                                   │
│  • 工作日白天高频使用手机处理消息                     │
│  • 习惯用微信作为主要工作沟通工具                     │
│  • 遇到问题先搜索，其次问同事                         │
└─────────────────────────────────────────────────────┘
```

### 用户画像注意事项

- **基于数据**，不是凭空想象（"假设型 Persona"是反模式）
- 一个产品通常有 2-4 个 Persona，聚焦核心用户
- 定期更新，随研究深入而修正

## 2.7 用户旅程地图（User Journey Map）

用户旅程地图可视化用户完成某个目标的全过程，找出痛点和机会点。

### 旅程地图结构

```
场景：用户首次使用外卖 App 点餐

阶段      │ 发现需求  │ 搜索选择  │ 下单支付  │ 等待收餐  │ 收餐评价
─────────┼──────────┼──────────┼──────────┼──────────┼─────────
用户行为  │ 感到饿   │ 浏览菜品  │ 加入购物车│ 查看进度  │ 给差评
         │ 打开 App │ 看评价    │ 填地址   │ 打电话    │
─────────┼──────────┼──────────┼──────────┼──────────┼─────────
想法/感受 │ 想吃什么 │ 哪家好？  │ 这么贵？  │ 怎么还没来│ 凉了!
         │ 好麻烦   │ 评价真假？│ 没零钱   │ 焦虑等待  │ 差体验
─────────┼──────────┼──────────┼──────────┼──────────┼─────────
情绪曲线  │  😐      │  😕       │  😤       │  😰       │  😡
─────────┼──────────┼──────────┼──────────┼──────────┼─────────
痛点      │ 选择困难 │ 信息不透明│ 支付繁琐  │ 时间不准确│ 投诉渠道难
─────────┼──────────┼──────────┼──────────┼──────────┼─────────
机会点    │ 智能推荐 │ 真实评价  │ 一键支付  │ 实时追踪  │ 快速补偿
```

## 2.8 How Might We（我们如何能……）

将痛点转化为设计机会的标准句式：

**格式**：`我们如何能够 [动词] [用户] 的 [需求/问题]，从而 [目标]？`

**示例**：
- 痛点：用户找不到想要的餐厅
- HMW：我们如何能帮助犹豫不决的用户，在 30 秒内找到满意的餐厅？

## 本章实践作业

**作业 1**：对 3 位朋友进行用户访谈（每人 15 分钟），主题：
> "你在工作/学习中是如何管理时间和任务的？"

记录要点：
- 他们用什么工具？为什么？
- 最大的困扰是什么？
- 他们理想的状态是什么？

**作业 2**：选择一个垂直领域（如：在线教育、健身 App、记账工具），对 3 款竞品进行分析，填写竞品对比矩阵。

**作业 3**：基于访谈结果，创建 1 个用户画像。
